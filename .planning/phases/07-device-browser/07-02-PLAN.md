Modify `AbletonMCP_Remote_Script/handlers/browser.py` with four changes:

**Change 1: get_browser_tree with max_depth**
In the `get_browser_tree` method:
- Extract `max_depth` from params: `max_depth = params.get("max_depth", 1) if isinstance(params, dict) else 1`
- Cap max_depth to a reasonable limit: `max_depth = min(max_depth, 5)` to prevent performance issues
- Modify the inner `process_item` function to accept `depth` and `max_depth` parameters:
  ```python
  def process_item(item, depth=0, max_depth=1):
      if not item:
          return None
      info = {
          "name": item.name if hasattr(item, "name") else "Unknown",
          "is_folder": hasattr(item, "children") and bool(item.children),
          "is_device": hasattr(item, "is_device") and item.is_device,
          "is_loadable": hasattr(item, "is_loadable") and item.is_loadable,
          "uri": item.uri if hasattr(item, "uri") else None,
          "children": [],
      }
      if depth < max_depth and hasattr(item, "children") and item.children:
          for child in item.children:
              child_info = process_item(child, depth + 1, max_depth)
              if child_info:
                  info["children"].append(child_info)
      return info
  ```
- Pass max_depth to all `process_item` calls: `process_item(app.browser.instruments, 0, max_depth)`

**Change 2: Track-type guard in _load_browser_item for instrument-on-audio-track**
In `_load_browser_item`, BEFORE the item lookup/load logic, add a track-type guard:
- After resolving the track (`track = self._song.tracks[track_index]`), check if the track is an audio track
- Determine if the item being loaded is an instrument. Use the resolved path or URI to check: if the root category is "instruments" or "drums" (from the path), or if the URI starts with an instrument-related prefix, treat it as an instrument load.
- For path-based detection: if `path` param is provided, check if `path.split("/")[0].lower()` is in `("instruments", "drums")`
- For URI-based detection: after finding the item, check `hasattr(item, "is_device") and item.is_device` (instruments are devices)
- The simplest reliable check: detect audio tracks via `hasattr(track, 'has_audio_input') and track.has_audio_input and not track.has_midi_input`. In Ableton's API, audio tracks have `has_audio_input=True` and `has_midi_input=False`.
- If track is an audio track AND the item path/category suggests an instrument (instruments, drums), raise ValueError with the EXACT message from user decision: `"Cannot load instrument on audio track. Use a MIDI track instead."`
- This guard runs BEFORE `load_item` is called, so the error is returned immediately.

**Change 3: Path-based loading in _load_browser_item**
In `_load_browser_item`, add path resolution before URI lookup:
- Extract `path` from params: `path = params.get("path", None)`
- If item_uri is empty/None but path is provided, resolve path to a browser item:
  - Split path on "/", get root category from _CATEGORY_MAP
  - Navigate children matching path parts (case-insensitive) -- same logic as get_browser_items_at_path
  - Once found, use the item directly (skip URI search)
  - Cache the resolved item in self._browser_path_cache using its URI
- If both item_uri and path are empty, raise ValueError("Provide item_uri or path")
- The rest of _load_browser_item proceeds unchanged with the resolved item

**Change 4: Include parameter list in load response**
In `_verify_load`, when load succeeds (devices_after_count > devices_before):
- Get the newly loaded device: `new_device = track.devices[-1]` (loaded device appears at end, or the new one is devices_after_count - 1 index)
- Actually, the new device might be anywhere depending on insertion. Safer: compare before/after. But simplest for first device: `new_device = track.devices[devices_after_count - 1]` or just get the last device.
- Build parameter list: `[{"index": i, "name": p.name, "value": p.value, "min": p.min, "max": p.max, "is_quantized": p.is_quantized} for i, p in enumerate(new_device.parameters)]`
- Add "parameters" key to the result dict

Do NOT remove `load_drum_kit` related commands from browser.py -- that will be handled in Plan 03 on the MCP tools side. The Remote Script handler `_get_browser_item`, `_get_browser_categories`, `_get_browser_items` can remain.
