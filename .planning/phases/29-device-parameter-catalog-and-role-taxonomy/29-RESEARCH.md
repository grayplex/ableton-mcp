# Phase 29: Device Parameter Catalog and Role Taxonomy - Research

**Researched:** 2026-03-28
**Domain:** Static device parameter catalog, unit conversion formulas, role taxonomy, bootstrap scripting
**Confidence:** HIGH

## Summary

Phase 29 creates a queryable static catalog of exact Ableton API parameter names, value ranges, and normalized-to-natural-unit conversion formulas for 12 built-in devices, plus a canonical role taxonomy for recipe lookup. The catalog must be bootstrapped from a live Ableton session -- never hand-authored from documentation.

The project already has a mature pattern for static data catalogs (the `MCP_Server/genres/` package with `catalog.py`, `schema.py`, and `__init__.py`), MCP tool registration (`@mcp.tool()` in `MCP_Server/tools/`), and Remote Script communication (`get_ableton_connection()` + `send_command()`). Phase 29 follows these patterns closely but is simpler: no auto-discovery (pkgutil), no alias resolution complexity -- just a static dict generated by a bootstrap script.

**Primary recommendation:** Follow the genres package pattern but simpler. Bootstrap script queries live Ableton via existing `get_device_parameters` command, writes `MCP_Server/devices/catalog.py`. Two MCP tools (`get_device_catalog`, `get_role_taxonomy`) in `MCP_Server/tools/catalog.py` read from the static module. Conversion formulas are structured dicts (no `eval()`), hand-curated after bootstrap captures raw parameter data.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01: Catalog Bootstrap Process** -- One-time bootstrap script runs against live Ableton, writes static Python file committed to repo. Uses existing `get_device_parameters` Remote Script command. Output goes to `MCP_Server/devices/catalog.py`.
- **D-02: Catalog Storage Format** -- Single `MCP_Server/devices/catalog.py` with `CATALOG` dict keyed by Ableton class name (e.g., `"Eq8"`, `"Compressor"`). Each entry holds display name, parameters list, and conversions. All 12 devices in one file.

### Pattern 1: Static Data Module (from genres/catalog.py)
**What:** Data lives in a Python module as a top-level constant dict. Tools import and query it. No database, no file I/O at runtime.
**When to use:** When data is curated/generated once and committed to the repo.
**Example:**
```python

    "Eq8": {
        "display_name": "EQ Eight",
        "parameters": [
            {
                "name": "1 Filter On A",
                "index": 1,
                "min": 0.0,
                "max": 1.0,
                "is_quantized": True,
                "conversion": None,
            },
            {
                "name": "1 Frequency A",
                "index": 2,
                "min": 0.0,
                "max": 1.0,
                "is_quantized": False,
                "conversion": {
                    "type": "log",
                    "natural_min": 20,
                    "natural_max": 22050,
                    "unit": "Hz",
                },
            },
            # ...
        ],
    },
    # ...
}

ROLES = [
    "kick", "bass", "lead", "pad",
    "chords", "vocal", "atmospheric",
    "return", "master",
]
```

### Pattern 2: MCP Tool Module (from tools/genres.py)
**What:** Thin wrapper that imports data from the catalog module and exposes it via `@mcp.tool()`. Returns JSON-serialized strings.
**When to use:** Every MCP-exposed query.
**Example:**
```python
# MCP_Server/tools/catalog.py
import json
from mcp.server.fastmcp import Context
from MCP_Server.connection import format_error
from MCP_Server.server import mcp
from MCP_Server.devices.catalog import CATALOG, ROLES

@mcp.tool()
def get_device_catalog(ctx: Context, device_name: str) -> str:
    """Get parameter catalog for a device. Accepts display name or class name."""
    # Lookup by class name first, then by display name
    entry = CATALOG.get(device_name)
    if entry is None:
        for class_name, data in CATALOG.items():
            if data["display_name"].lower() == device_name.lower():
                entry = data
                break
    if entry is None:
        return format_error(
            f"Device '{device_name}' not found in catalog",
            suggestion="Use exact device name: EQ Eight, Compressor, etc.",
        )
    return json.dumps(entry, indent=2)

@mcp.tool()
def get_role_taxonomy(ctx: Context) -> str:
    """Get the canonical mixing role taxonomy for recipe lookup."""
    return json.dumps({"roles": ROLES})
```

### Pattern 3: Bootstrap Script (new pattern)
**What:** Script connects to live Ableton, queries each target device, writes Python source file.
**When to use:** One-time catalog generation or refresh.
**Key detail:** The bootstrap script must have a way to locate each of the 12 target devices in the live session. Two approaches:
1. **Require a prepared session** -- user opens a session with all 12 devices loaded on known tracks
2. **Load devices programmatically** -- script uses `load_instrument_or_effect` to place each device, then queries it

Approach 1 is simpler and more reliable. The bootstrap script can accept a JSON config mapping device display names to track/device indices, or simply iterate all tracks looking for devices by class name.

### Pattern 4: Tool Registration (from tools/__init__.py)
**What:** New tool module must be imported in `MCP_Server/tools/__init__.py` for `@mcp.tool()` decorators to fire.
**Example:**
```python
# MCP_Server/tools/__init__.py
from . import ..., catalog  # noqa: F401
```

### Anti-Patterns to Avoid
- **Hand-authoring parameter names from documentation** -- Parameter names and ranges MUST come from live Ableton `get_device_parameters` output. Documentation may differ from what the API actually returns.
- **Using eval() for conversion formulas** -- Conversions are structured dicts, not string expressions. Runtime code applies the formula based on `type` field.
- **Putting conversion logic in the catalog** -- The catalog stores conversion metadata (type, range, unit). The actual conversion function lives in the consumer code (Phase 31 apply tools).
- **Over-engineering the bootstrap** -- It runs once, maybe again after an Ableton update. Keep it simple: connect, query, write file.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Ableton connection | Custom socket code | `get_ableton_connection()` from `MCP_Server.connection` | Already handles reconnection, timeouts, framing |
| Device parameter query | Direct LOM access | `send_command("get_device_parameters", ...)` | Existing Remote Script handler returns `{name, value, min, max, is_quantized}` |
| MCP tool registration | Manual tool setup | `@mcp.tool()` decorator from FastMCP | Project-wide pattern, auto-registers with server |
| Error formatting | Ad-hoc error strings | `format_error(message, detail, suggestion)` | Consistent AI-readable error format |
| Parameter name discovery | Reading Ableton docs | Bootstrap from live session | Docs may not match actual API parameter names |

## Common Pitfalls

### Pitfall 1: Ableton Class Names vs Display Names
**What goes wrong:** Assuming the class name matches the display name (e.g., "EQ Eight" vs "Eq8", "Glue Compressor" vs "GlueCompressor").
**Why it happens:** The Remote Script returns `class_name` from `device.class_name` which is an internal Ableton identifier, not the user-facing name.
**How to avoid:** The bootstrap script captures both `device.name` (display) and `device.class_name` (internal). The catalog keys on `class_name` and includes `display_name` for human lookup.
**Warning signs:** `get_device_catalog("EQ Eight")` returning "not found" because the key is "Eq8".

### Pitfall 2: Normalized vs Natural Parameter Values
**What goes wrong:** Assuming all parameter values are in human-readable units. Many Ableton parameters store normalized 0.0-1.0 values that represent logarithmic scales (frequency) or dB ranges.
**Why it happens:** The `get_device_parameters` handler returns `min`/`max` as the API sees them (often 0.0 to 1.0), not the display values.
**How to avoid:** The conversion dict on each parameter entry captures the mapping. For Phase 29, populate conversion metadata for known parameter types (frequency, gain, time). Parameters without clear conversion get `None`.
**Warning signs:** Setting a frequency parameter to 440 when the API expects 0.0-1.0.

### Pitfall 3: Parameter Index 0 is Always "Device On"
**What goes wrong:** Including "Device On" as a meaningful parameter in the catalog, or treating index 0 as a sound-shaping parameter.
**Why it happens:** Every Ableton device has parameter index 0 as the on/off toggle.
**How to avoid:** Document this in the catalog (include it but note it's the device enable toggle). Downstream recipe code should skip index 0.
**Warning signs:** Recipes trying to set "Device On" to a frequency value.

### Pitfall 4: Bootstrap Requires All 12 Devices Loaded
**What goes wrong:** Running the bootstrap script with a session that's missing some target devices.
**Why it happens:** The script needs to iterate tracks and find devices by class name or have them at known positions.
**How to avoid:** Either require a "catalog session" file with all 12 devices pre-loaded, or have the script report which devices it found vs. missed. The bootstrap script should exit with clear error if any of the 12 are missing.
**Warning signs:** Catalog has fewer than 12 entries after bootstrap.

### Pitfall 5: Tool Module Not Registered
**What goes wrong:** New `MCP_Server/tools/catalog.py` created but tools don't appear in MCP.
**Why it happens:** The module isn't imported in `MCP_Server/tools/__init__.py`, so `@mcp.tool()` decorators never fire.
**How to avoid:** Add `catalog` to the import list in `__init__.py`.
**Warning signs:** Tool calls return "unknown tool" errors.

### Pitfall 6: Conversion Formulas Need Manual Curation
**What goes wrong:** Expecting the bootstrap script to automatically determine conversion formulas.
**Why it happens:** Ableton's `get_device_parameters` returns `{name, value, min, max, is_quantized}` but NOT the natural unit range or scale type. The API has no "this parameter is logarithmic Hz" metadata.
**How to avoid:** Bootstrap captures the raw parameter data. Conversion metadata is added manually based on known Ableton parameter behaviors (EQ frequency is always log 20-22050 Hz, gain params are dB, etc.). Document which conversions are hand-curated vs. bootstrapped.
**Warning signs:** All conversion fields set to `None` with no manual curation step.
